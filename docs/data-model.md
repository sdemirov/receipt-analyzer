# Data model

The single source of truth for the apps is **`data/receipts.db`** (SQLite),
generated by `extract/build_db.py`. Two CSVs are editable inputs that feed it.

## SQLite schema

### `receipts`
One row per unique receipt (deduped by `unp`).

| Column | Type | Notes |
|--------|------|-------|
| `id` | INTEGER PK | |
| `unp` | TEXT UNIQUE | Unique receipt number (dedupe key) |
| `purchase_date` | TEXT | ISO `YYYY-MM-DD`, parsed from the receipt body |
| `purchase_time` | TEXT | `HH:MM:SS` |
| `branch_id` | TEXT | Store code, e.g. `6500` |
| `store_name` | TEXT | Full store name |
| `subtotal` | REAL | Before discounts (falls back to total) |
| `total` | REAL | Amount paid |
| `card_savings` | REAL | Loyalty-card savings |
| `promo_savings` | REAL | Item-level price-promo savings |
| `payment_method` | TEXT | `В брой` / `Кредитна карта` / `Дебитна карта` |
| `points` | INTEGER | Loyalty points earned |
| `source_pdf` | TEXT | Originating path **relative to `RECEIPTS_DIR`**, e.g. `Kaufland/2026-05-30.pdf` or `Lidl/Файл_000 (1).png` |
| `raw_text` | TEXT | Full extracted receipt text (for the modal) |

### `receipt_pdfs`
Stores the **original receipt bytes** (PDF or PNG) so the app is self-contained (no
dependency on the source folder at runtime). Kept in a separate table so analytics
queries never load blobs.

| Column | Type | Notes |
|--------|------|-------|
| `receipt_id` | INTEGER PK → receipts.id | |
| `pdf` | BLOB | original PDF **or PNG** bytes |
| `media_type` | TEXT | `application/pdf` (Kaufland) or `image/png` (Lidl) |

`GET /receipts/{id}/pdf` serves the blob with its stored `media_type`. The React
modal detects `source_pdf.endsWith(".png")` to choose between an `<img>` preview
("Снимка" tab) and an `<iframe>` ("PDF" tab).

### `products`
One row per distinct product (after fuzzy grouping).

| Column | Type | Notes |
|--------|------|-------|
| `id` | INTEGER PK | `product_id` |
| `canonical_name` | TEXT | Display name (most frequent raw spelling) |
| `brand` | TEXT | Auto-suggested, editable (may be empty) |
| `category` | TEXT | Auto-suggested, editable (may be empty) |
| `search_key` | TEXT | Latin phonetic skeleton of the name + raw variants, for bilingual (BG↔EN) search ([`translit.py`](../translit.py)) |

### `line_items`
One row per purchased line.

| Column | Type | Notes |
|--------|------|-------|
| `id` | INTEGER PK | |
| `receipt_id` | INTEGER FK → receipts.id | |
| `product_id` | INTEGER FK → products.id | |
| `raw_name` | TEXT | Exact name printed on the receipt |
| `qty` | REAL | Pieces, or **kilograms** for weighed items |
| `unit_price` | REAL | Per piece, or **per kg** for weighed items |
| `line_total` | REAL | |
| `vat_class` | TEXT | `А` / `Б` / `Г` |
| `vat_rate` | REAL | 0 / 20 / 9 |
| `unit_measure` | TEXT | `pc` or `kg` |
| `on_promo` | INTEGER | 1 if bought on a price promotion (`Вие спестявате` line) |
| `promo_saving` | REAL | amount saved on this line vs the regular price |

Indexes: `line_items(product_id)`, `line_items(receipt_id)`, `receipts(purchase_date)`.

### Entity relationships

```
receipts 1 ──< line_items >── 1 products
```

`unit_price` semantics depend on `unit_measure`: compare a product only with
itself (its measure is consistent), which the price chart does by grouping on
`product_id`.

## Editable CSV inputs

Both live in `data/`, are UTF-8, and are **regenerated-safe**: re-running
`build_db` keeps your edits and only appends newly-seen rows.

### `product_mapping.csv` — which raw names form a product
```
raw_name,canonical_name,product_id
Кроасан нуга,Кроасан нуга 90г,4
Кроасан нуга 90г,Кроасан нуга 90г,4
```
Edit `canonical_name` / `product_id` to merge or split products. Details and the
exact merge/split recipe: [product-matching.md](product-matching.md).

### `product_meta.csv` — display name, brand & category per product
```
product_id,canonical_name,display_name,brand,category
8,"Балкан Био КМ 3,6%","Балкан Био кисело мляко 3,6%",Балкан,Мляко и млечни
```
The join key is `product_id`. `canonical_name` is the **auto** name (read-only
reference). Set **`display_name`** to rename a product everywhere — it becomes
the shown name and is folded into `search_key` (the original short form stays
searchable too). Blank `display_name` = use the auto name. Edit `brand` /
`category` likewise. Edit in the app's **✏️ Преименуване** tab (which writes this
file and applies it live via `PUT /products/{id}/name`) or by hand; either way
re-running `build_db` keeps it. Details: [brand-category.md](brand-category.md).

## How edits survive rebuilds

- `normalize.build_mapping()` loads existing `product_mapping.csv`, preserves
  every row, and only clusters raw names not already mapped.
- `build_db.build_meta()` loads existing `product_meta.csv` and reuses its
  brand/category for known `product_id`s; only new products get fresh guesses.

> Caveat: deleting a CSV regenerates it from scratch and may renumber
> `product_id`s. Don't delete them once you've started editing — edit in place.

## Rebuilding

```bash
# Full build (Kaufland PDFs + Lidl PNGs) — requires Docker (tesseract is in the image)
docker compose run --rm app python -m extract.build_db

# Kaufland-only build on the host venv (no tesseract needed)
venv/Scripts/python.exe -m extract.build_db
```

Drops and recreates all four tables every run; the CSVs persist.
